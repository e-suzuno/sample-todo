# コミットメッセージ生成スキル

ステージ済みのgit変更を分析し、プロジェクトのコミット規約に沿ったメッセージを生成する。

## コミットメッセージのフォーマット

```
#<Issue番号> <type>: <日本語の説明>

<本文: 変更の理由・詳細>
```

- Issue番号がない場合は `<type>: <日本語の説明>` のみ（`#` なし）
- 本文は空行を挟んで記述し、変更の「なぜ」を中心に書く

## type 一覧

| type | 用途 | 例 |
|------|------|-----|
| `feat` | 新機能追加 | `#12 feat: ファイルアップロード用Presigned URL取得API追加` |
| `fix` | バグ修正 | `#15 fix: OTPロック時間の計算誤り修正` |
| `docs` | ドキュメント変更 | `#8 docs: シーケンス図にダウンロード通知フロー追加` |
| `refactor` | リファクタリング | `#20 refactor: Lambda共通エラーハンドリングを抽出` |
| `test` | テスト追加・修正 | `#12 test: 共有作成APIのユニットテスト追加` |
| `chore` | ビルド・設定変更 | `#5 chore: Terraform state backend設定追加` |
| `ci` | CI/CD設定 | `#3 ci: GitHub Actions デプロイワークフロー追加` |

## 生成手順

### 1. 変更内容の把握

以下のコマンドを実行して情報を収集する:

```bash
# ステージ済み変更の確認
git diff --staged

# ブランチ名の確認（Issue番号の推測に使う）
git branch --show-current

# 直近のコミット履歴（文体の参考）
git log --oneline -5
```

### 2. type の判定

変更内容から最も適切なtypeを1つ選ぶ。判定の指針:

- **新しいファイル・関数・エンドポイントの追加** → `feat`
- **既存の挙動の不具合を直している** → `fix`
- **`.md` ファイルのみ、またはコメント・ドキュメントの変更** → `docs`
- **動作を変えずにコード構造を改善** → `refactor`
- **テストファイルの追加・修正が主** → `test`
- **設定ファイル・依存関係の変更** → `chore`
- **CI/CDパイプラインの変更** → `ci`

複数のtypeにまたがる変更がある場合は、最も主要な変更のtypeを選ぶ。
ただし、本来は1コミット=1論理変更が原則なので、分割を提案することも検討する。

### 3. Issue番号の推測

ブランチ名からIssue番号を推測する:

- `feature/#12-upload-api` → `#12`
- `feature/12-upload-api` → `#12`
- `fix/#45-otp-bug` → `#45`

ブランチ名に番号がない場合（`feature/add-logging` や `main` など）はIssue番号なしで生成する。
推測に自信がない場合はユーザーに確認する。

### 4. 説明文の作成

- **1行目（サマリー）**: 変更内容を簡潔に日本語で記述（50文字以内を目安）
- **本文**: 空行の後に、変更の理由や背景を1〜3文で記述

説明文のポイント:
- 「何を」ではなく「何のために」を意識する
- 技術的な詳細よりも、変更の意図を伝える
- 冗長にならず、必要十分な情報量にする

### 5. メッセージの提案

生成したメッセージをユーザーに提示し、確認を求める。
ユーザーが承認したら、heredoc形式でコミットを実行する:

```bash
git commit -m "$(cat <<'EOF'
#12 feat: ファイルアップロード用Presigned URL取得API追加

S3への直接アップロードを可能にするため、署名付きURL生成エンドポイントを追加。
管理者がファイルを登録する際のアップロードフローで使用する。
EOF
)"
```

## 注意事項

- ユーザーの明示的な承認なしにコミットを実行しない
- ステージされていない変更がある場合は、何がステージ済みで何がそうでないかを伝える
- 複数の論理的変更がステージされている場合は、分割コミットを提案する
- WIPコミットは避ける。やむを得ない場合は `chore: WIP <説明>` とする
