# プルリクエスト作成スキル

ブランチの差分を分析し、Conventional Commits規約に沿ったPRタイトルと本文を生成して
プルリクエストを作成する。

## PRタイトルのフォーマット

```
#<Issue番号> <type>: <日本語の説明>
```

- コミットメッセージと同じフォーマットを使用する
- Issue番号がない場合は `<type>: <日本語の説明>` のみ（`#` なし）

### type 一覧

| type | 用途 |
|------|------|
| `feat` | 新機能追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメント変更 |
| `refactor` | リファクタリング |
| `test` | テスト追加・修正 |
| `chore` | ビルド・設定変更 |
| `ci` | CI/CD設定 |

## PR本文のフォーマット

```markdown
## 概要

<!-- このPRで何をしたか・なぜ必要かを2〜3文で記述 -->

## 変更内容

<!-- 主な変更点を箇条書きで記述 -->
-
-

## テスト計画

<!-- 変更内容に応じたテスト計画を記述（下記の判断基準を参照） -->

## 確認事項

<!-- レビュアーに特に見てほしい点・懸念点があれば記述 -->
-

## 関連Issue

<!-- Issue番号があれば記述（Closes #XX で自動クローズ） -->
Closes #
```

## テスト計画の判断基準

変更内容を分析し、以下の基準でテスト計画のレベルを決定する。

### レベル1: 軽量（確認した操作を箇条書き）

**対象**: `docs` / `chore` / `ci` / 設定ファイルのみの変更

```markdown
## テスト計画

- ドキュメントの表示確認
- リンク切れがないことを確認
```

### レベル2: 標準（テスト項目・手順・期待結果を記述）

**対象**: `refactor` / `test` / 既存機能への軽微な変更

```markdown
## テスト計画

| 項目 | 手順 | 期待結果 |
|------|------|----------|
| 既存機能の動作確認 | 1. XXXを実行する | 従来と同じ結果が返る |
| エラーハンドリング | 1. 不正な値を入力する | エラーメッセージが表示される |
```

### レベル3: 詳細（項目・手順・期待結果＋影響範囲）

**対象**: `feat` / `fix` / APIや認証など影響範囲が広い変更

```markdown
## テスト計画

### 動作確認

| 項目 | 手順 | 期待結果 |
|------|------|----------|
| 正常系: XXX | 1. YYYを実行する 2. ZZZを確認する | 期待する結果 |
| 異常系: XXX | 1. 不正な値を入力する | エラーが適切に返る |

### 影響範囲の確認

- [ ] 既存の〇〇機能に影響がないこと
- [ ] 〇〇APIのレスポンスが変わっていないこと

### 非機能確認（必要な場合）

- [ ] パフォーマンスの劣化がないこと
- [ ] ログが適切に出力されること
```

## 生成手順

### 1. 利用可能なツールの確認

最初に使用できるツールを確認し、以降の手順で適切な方法を選択する:

```bash
# gh コマンドの確認
gh --version
```

- `gh` コマンドが使える → **方法A**
- `mcp__github__*` ツールが使える → **方法B**
- どちらも使えない → **方法C**

### 2. 情報収集

以下のコマンドを実行して情報を収集する:

```bash
# 現在のブランチ名確認
git branch --show-current

# マージ先ブランチの確認（通常はmainまたはdevelop）
git remote show origin | grep "HEAD branch"

# ブランチ間の差分コミット一覧
git log main..HEAD --oneline

# ブランチ間のファイル差分
git diff main...HEAD --stat

# 詳細な差分（必要に応じて）
git diff main...HEAD
```

### 3. タイトルの生成

- ブランチ名からIssue番号を推測する
    - `feature/#12-upload-api` → `#12`
    - `feature/12-upload-api` → `#12`
    - `fix/#45-otp-bug` → `#45`
- 差分コミット群から最も主要な変更のtypeを選ぶ
- 変更内容を50文字以内の日本語で要約する

### 4. 本文の生成

- **概要**: 変更の目的・背景を2〜3文で記述
- **変更内容**: 差分から主要な変更点を箇条書きで抽出
- **テスト計画**: 上記の判断基準に従いレベルを選択して記述
- **確認事項**: レビュアーへの注意点・懸念点があれば記述。なければ省略可
- **関連Issue**: Issue番号がある場合は `Closes #XX` で記述

### 5. ユーザーへの確認

生成したタイトルと本文をユーザーに提示し、確認を求める。
修正が必要であれば対応してから次のステップへ進む。

### 6. PRの作成

ユーザーが承認したら、利用可能なツールに応じて以下の方法でPRを作成する。

#### 方法A: gh コマンド

```bash
gh pr create \
  --title "#12 feat: ファイルアップロード用Presigned URL取得API追加" \
  --body "$(cat <<'EOF'
## 概要

S3への直接アップロードを実現するため、署名付きURL生成エンドポイントを追加した。
管理者がファイルを登録する際のアップロードフローで使用する。

## 変更内容

- Presigned URL生成Lambda関数を追加
- API Gatewayにエンドポイントを追加
- S3バケットポリシーを更新

## テスト計画

### 動作確認

| 項目 | 手順 | 期待結果 |
|------|------|----------|
| 正常系: URL取得 | 1. APIを呼び出す 2. レスポンスを確認する | 署名付きURLが返る |
| 異常系: 権限なし | 1. 権限のないユーザーで呼び出す | 403エラーが返る |

### 影響範囲の確認

- [ ] 既存のファイルダウンロード機能に影響がないこと
- [ ] S3バケットへの既存アクセスが変わっていないこと

## 確認事項

- URLの有効期限を15分に設定しているが、要件に合っているか確認をお願いしたい

## 関連Issue

Closes #12
EOF
)" \
  --base main
```

#### 方法B: GitHub MCP Server

`mcp__github__create_pull_request` ツールを使用する:

- `title`: 生成したPRタイトル
- `body`: 生成したPR本文
- `head`: 現在のブランチ名
- `base`: マージ先ブランチ名（通常 `main` または `develop`）

#### 方法C: GitHub REST API（gh・MCP どちらも使えない場合）

```bash
curl -X POST https://api.github.com/repos/{owner}/{repo}/pulls \
  -H "Authorization: token $GITHUB_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "title": "#12 feat: ファイルアップロード用Presigned URL取得API追加",
    "body": "## 概要\n\n...",
    "head": "feature/12-upload-api",
    "base": "main"
  }'
```

`GITHUB_TOKEN` が未設定の場合はユーザーに確認する。

## 注意事項

- ユーザーの明示的な承認なしにPRを作成しない
- リモートにpush済みかを確認する。未pushの場合はpushを提案する:
  ```bash
  git push -u origin $(git branch --show-current)
  ```
- Draft PRにしたい場合:
    - 方法A: `--draft` フラグを追加
    - 方法B: `draft: true` を指定
- マージ先ブランチが不明な場合はユーザーに確認する
- 複数の論理的変更がブランチに含まれる場合は、PRの分割を提案する
