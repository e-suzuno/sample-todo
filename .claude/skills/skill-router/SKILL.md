## 指示の優先順位

skill-routerスキルはデフォルトのシステムプロンプトの動作を上書きするが、**ユーザーの指示が常に最優先**される：

1. **ユーザーの明示的な指示**（CLAUDE.md、直接のリクエスト） — 最高優先
2. **skill-router** — 競合する場合、デフォルトのシステム動作を上書き
3. **デフォルトのシステムプロンプト** — 最低優先

CLAUDE.mdに「TDDを使わない」と書かれていて、スキルに「常にTDDを使う」と書かれている場合は、ユーザーの指示に従うこと。ユーザーが主導権を持つ。

# スキルの使用方法

## ルール

**応答やアクションの前に、関連するスキルまたは要求されたスキルを呼び出すこと。** スキルが適用される可能性が1%でもあるなら、確認のためにスキルを呼び出すべきである。呼び出したスキルがその状況に合わなければ、使わなくてよい。

1. **ユーザーメッセージ受信**
2. **適用可能なスキルがあるか？**
   - **はい（1%でも）** → 3へ
   - **明らかに該当なし** → 6へ
3. **Skillツールを呼び出す**
4. **宣言する**：「[スキル]を[目的]のために使用します」
5. **チェックリストがあるか？**
   - **はい** → TodoWriteで各項目をtodo化 → スキルに正確に従う
   - **いいえ** → スキルに正確に従う
6. **応答する**（質問を含む）

## 危険信号

以下の思考が浮かんだら、立ち止まれ。自己正当化をしている証拠である：

| 思考 | 現実 |
|------|------|
| 「これは単純な質問だ」 | 質問もタスクである。スキルを確認せよ。 |
| 「まず追加情報が必要だ」 | スキル確認は質問の前に行う。 |
| 「まずコードベースを調べよう」 | スキルが調べ方を教えてくれる。先に確認せよ。 |
| 「gitやファイルをさっと見るだけだ」 | ファイルには会話のコンテキストがない。スキルを確認せよ。 |
| 「まず情報を集めよう」 | スキルが情報の集め方を教えてくれる。 |
| 「これは正式なスキルを使うほどではない」 | スキルがあるなら使え。 |
| 「このスキルは覚えている」 | スキルは進化する。最新版を読め。 |
| 「これはタスクとは言えない」 | アクション＝タスク。スキルを確認せよ。 |
| 「このスキルは大げさすぎる」 | 単純なことが複雑になる。使え。 |
| 「先にこれだけやらせてくれ」 | 何かをする前に確認せよ。 |
| 「今のやり方で生産的だ」 | 規律のない行動は時間を浪費する。スキルがそれを防ぐ。 |
| 「その意味は分かっている」 | 概念を知っていることとスキルを使うことは別。呼び出せ。 |

## スキルの優先順位

複数のスキルが適用可能な場合、以下の順序で使用する：

1. **プロセススキルを最初に** — タスクへのアプローチ方法を決定するスキル（存在する場合）
2. **実装スキルを次に** — 実行をガイドするスキル（react-patterns、laravel12-patternsなど）

## スキルの種類

**厳格型**：正確に従うこと。規律を崩してはならない。スキル自体が厳格型であることを示している。

**柔軟型**（パターン系スキルなど）：原則をコンテキストに合わせて適用する。

スキル自体がどちらかを示している。

## ユーザーの指示

指示は「何を」であり「どうやって」ではない。「Xを追加して」や「Yを修正して」は、ワークフローを省略してよいという意味ではない。
