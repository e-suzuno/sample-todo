# テスト駆動開発（TDD）ワークフロー

このスキルはTDDプロセスの管理に特化する。テストコードの具体的な書き方（テストランナーの設定、モックパターン、セレクタの使い方等）はこのスキルの対象外である。プロジェクトで使用しているフレームワーク向けのスキルがあれば、そちらの `references/testing-patterns.md` 等を併用すること。

## リファレンス

| 依頼のキーワード | リファレンス |
|---|---|
| テスト要件、カバレッジ基準、AAAパターン、トラブルシューティング | `references/testing-requirements.md` |

## 発動条件

- 新機能や機能追加の実装
- バグ修正
- 既存コードのリファクタリング
- APIエンドポイントの追加
- 新規コンポーネントの作成

## 基本原則

### 1. テストを先に書く

**必ず**テストを先に書き、そのテストを通すための実装コードを書く。

### 2. カバレッジ要件

- 最低80%のカバレッジ（ユニット + インテグレーション + E2E）
- エッジケースを網羅する
- エラーシナリオをテストする
- 境界条件を検証する

### 3. テスト種別

| 種別 | 対象 | 必須度 |
|------|------|--------|
| ユニットテスト | 個別の関数・ユーティリティ・ロジック | 常に必須 |
| インテグレーションテスト | APIエンドポイント・DB操作・サービス連携 | 常に必須 |
| E2Eテスト | クリティカルなユーザーフロー | 重要なパスで必須 |

具体的なテストコードの記述方法は、プロジェクトで使用しているフレームワーク向けスキルがあればそちらを参照すること。

### 4. Gitチェックポイント

リポジトリがGit管理下にある場合、TDD各フェーズ完了時にチェックポイントコミットを作成する。

- チェックポイントコミットは完了前にスカッシュやリライトしない
- コミットメッセージは **プロジェクトのコミット規約に従う**こと（コミットメッセージ生成スキルがあればそちらを使用する）
- TDD各フェーズに対応するtypeガイド:
  - **RED（テスト作成）** → `test:` を使用
  - **GREEN（実装）** → `fix:` または `feat:` を使用
  - **IMPROVE（リファクタリング）** → `refactor:` を使用
- 推奨コンパクトワークフロー:
  - REDの検証を含むテスト追加コミット 1つ
  - GREENの検証を含む実装コミット 1つ
  - リファクタリングコミット 1つ（任意）

## TDDワークフロー手順

### Step 1: ユーザージャーニーを書く

```
ロール: [役割]として
目的: [アクション]したい
理由: [メリット]を得るため

例:
ロール: ユーザーとして
目的: マーケットをセマンティック検索したい
理由: 正確なキーワードがなくても関連するマーケットを見つけるため
```

### Step 2: テストケースを生成する

各ユーザージャーニーに対して、網羅的なテストケースを作成する:

- 正常系（ハッピーパス）
- 空の入力
- エラー・フォールバック
- ソート・フィルタリング

具体的なテストコードの記述方法は、プロジェクトで使用しているフレームワーク向けスキルがあればそちらを参照。

### Step 3: テストを実行する — 失敗を確認（RED）

```bash
# テストランナーのコマンドはプロジェクトに応じて変更
npm test        # JavaScript/TypeScript
php artisan test  # Laravel
```

このステップは必須であり、すべてのプロダクションコード変更に対するREDゲートとなる。

プロダクションコードを編集する前に、有効なRED状態を以下のいずれかで確認する必要がある:

- **ランタイムRED**:
  - テストターゲットのコンパイルが成功する
  - 新規または変更したテストが実際に実行される
  - 結果がREDである
- **コンパイルタイムRED**:
  - 新規テストがバグのあるコードパスを新たに参照・実行する
  - コンパイル失敗自体が意図されたREDシグナルである
- いずれの場合も、失敗は意図したビジネスロジックのバグ、未定義の振る舞い、または未実装によるものであること
- 無関係な構文エラー、テストセットアップの破損、依存関係の欠落による失敗はRED状態として認めない

書いただけでコンパイル・実行されていないテストはRED状態とは認めない。

RED状態の確認が完了するまで、プロダクションコードを編集しないこと。

**Git管理下の場合**: RED検証後、直ちにチェックポイントコミットを作成する。

### Step 4: 最小限の実装を書く（GREEN）

テストを通すための**最小限の**コードだけを書く。

**Git管理下の場合**: 実装をステージするが、コミットはStep 5のGREEN検証後まで待つ。

### Step 5: テストを再実行する — 成功を確認（GREEN）

```bash
npm test        # JavaScript/TypeScript
php artisan test  # Laravel
```

修正後に同じテストターゲットを再実行し、以前失敗していたテストがGREENになったことを確認する。

有効なGREEN結果が得られるまで、リファクタリングに進んではならない。

**Git管理下の場合**: GREEN検証後、直ちにチェックポイントコミットを作成する。

### Step 6: リファクタリング（IMPROVE）

テストがグリーンを維持したまま、コード品質を改善する:

- 重複の除去
- 命名の改善
- パフォーマンスの最適化
- 可読性の向上

**Git管理下の場合**: リファクタリング完了後、テストがグリーンであることを確認してからチェックポイントコミットを作成する。

### Step 7: カバレッジを確認する

```bash
# カバレッジレポートの生成（プロジェクトに応じてコマンドを変更）
npm run test:coverage
php artisan test --coverage --min=80
```

80%以上のカバレッジが達成されていることを確認する。

## テストのアンチパターン

### 避けるべき: 実装の詳細をテストする
内部状態ではなく、外部から観察可能な振る舞いをテストする。

### 避けるべき: テスト間の依存
テストは独立して実行できるようにする。共有状態を持たない。

### 避けるべき: 検証の不足
何も検証しないテスト（通るだけで意味がない）を書かない。

### 避けるべき: 外部依存のモック漏れ
外部サービスへの依存は適切にモックする。

## 継続的テスト

### 開発中のウォッチモード
```bash
npm test -- --watch
# ファイル変更時に自動でテストが実行される
```

### プリコミットフック
```bash
# コミット前に自動実行
npm test && npm run lint
```

### CI/CDとの統合
```yaml
# GitHub Actions の例
- name: Run Tests
  run: npm test -- --coverage
- name: Upload Coverage
  uses: codecov/codecov-action@v3
```

## ベストプラクティス

1. **テストを先に書く** — 常にTDD
2. **1テスト1アサート** — 単一の振る舞いに集中する
3. **テスト名は説明的に** — テスト対象を明確にする
4. **Arrange-Act-Assert** — 明確なテスト構造
5. **外部依存をモックする** — ユニットテストを分離する
6. **エッジケースをテストする** — null, undefined, 空, 大量データ
7. **エラーパスをテストする** — ハッピーパスだけでなく
8. **テストは高速に** — ユニットテストは1件50ms以内を目安
9. **テスト後に片付ける** — 副作用を残さない
10. **カバレッジレポートを確認する** — ギャップを特定する

## 成功指標

- 80%以上のコードカバレッジ達成
- すべてのテストが成功（グリーン）
- スキップまたは無効化されたテストがない
- 高速なテスト実行（ユニットテストは30秒以内）
- E2Eテストがクリティカルなユーザーフローをカバー
- テストがプロダクション前にバグを検出している

---

**注意**: テストはオプションではない。自信を持ったリファクタリング、迅速な開発、プロダクションの信頼性を支える安全ネットである。
