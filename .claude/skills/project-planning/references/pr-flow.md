# コミットからデリバリーまでのフロー

フェーズ 6（コミット）とフェーズ 7（デリバリー）の詳細手順。

## フェーズ 6: コミット

全変更を Conventional Commits 規約に従ってコミットする。

### 手順

**`git-commitmsg` スキルの手順に従って**コミットメッセージを生成・実行する。
以下は project-planning スキル固有の追加ルール。

### コミット単位の判断

| 状況 | コミット戦略 |
|---|---|
| 単一の論理的変更 | 1 コミット |
| TDD フェーズが明確に分かれる | フェーズごとに分割（`test:` → `feat:` / `fix:` → `refactor:`） |
| 複数の機能にまたがる変更 | 機能ごとに分割 |

### 承認ゲート ②

コミット内容をユーザーに提示する:

- ステージ対象のファイル一覧
- コミットメッセージ案
- **ユーザーの承認後にコミットを実行する**

### Git 操作の失敗対応

```
git 操作の失敗
  ├── merge conflict → ユーザーに報告し判断を仰ぐ
  ├── push 拒否 → リモートの最新を取得してリベースを提案
  └── gh CLI 不可 → GitHub 利用不可パスに切り替える
```

## フェーズ 7: デリバリー

変更をリモートに push し、PR を作成する。

### GitHub 利用可能な場合

1. `git push -u origin <ブランチ名>` でリモートに push
2. **`git-pull-request` スキルの手順に従って**PR タイトル・本文を生成する
3. **承認ゲート ③**: PR 内容をユーザーに提示し、承認後に PR を作成する

### GitHub 利用不可の場合

- 変更内容のサマリーをユーザーに報告して完了とする
- `.plan/` の計画ファイルに結果を追記する

## 完了報告のフォーマット

PR 作成後（または GitHub 利用不可時のサマリー報告時）、以下の情報をユーザーに提示する:

```markdown
## タスク完了報告

**ブランチ**: <branch-name>
**PR**: <PR URL>（GitHub 利用時）
**Issue**: #<番号>（あれば）

### 完了したこと
- [達成事項 1]
- [達成事項 2]

### テスト結果
- ユニットテスト: PASS（X / X）
- カバレッジ: XX%

### レビュー結果
- 🔴 ブロッカー: 0 件
- 🟡 提案: X 件（対応済み / 次回対応）
- 💭 細かい指摘: X 件

### 残課題 / 次回タスク候補
- [スコープ外で記録した項目 1]
- [スコープ外で記録した項目 2]
```

## コミュニケーションスタイル

各フェーズの開始・完了時に進捗を報告する。報告の例:

- **フェーズ遷移**: 「フェーズ 1（戦略策定）を完了しました。フェーズ 2（計画作成）に進みます」
- **エージェント検出結果**: 「利用可能エージェント: テスト担当=[検出名], 実装担当=[検出名], レビュー担当=[検出名]」
- **委任**: 「テスト担当（[検出名]）にテスト作成を委任します」
- **結果サマリー**: 「レビュー結果: 🔴 0 件、🟡 2 件、💭 1 件。ブロッカーなしのためフェーズ 6 に進みます」
- **承認要求**: 「以下の計画で進めてよいですか？」
- **問題報告**: 「3 回のイテレーションでブロッカーが解消できていません。状況を報告します」
