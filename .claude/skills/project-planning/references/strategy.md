# フェーズ 1: 戦略策定（Strategy）

タスクを分析し、実行可能な戦略を策定するフェーズ。

## 目的

- ユーザー依頼のゴールを明確にする
- プロジェクトの技術スタックと利用可能エージェントを特定する
- 必要なら調査を行い、戦略を確定する
- 失敗シナリオを先に潰す（プレモーテム）

## ステップ

### 1. タスクの理解

- ユーザーの依頼文を分析し、ゴールを明確にする
- 不明点があればユーザーに質問する（先に進まない）

### 2. プロジェクト検出

開発開始前に以下を確認する:

| 観点 | 確認方法 |
|---|---|
| フレームワーク・言語 | `package.json` / `composer.json` / `pyproject.toml` / `go.mod` を Read |
| Git 環境 | `git-setup` スキルのステップ 1 に従い、GitHub 有無・デフォルトブランチ・`gh` CLI を確認 |
| ビルド・テストランナー | 上記マニフェストの `scripts` セクションや CI ファイルを確認 |
| 既存テストの場所 | `Glob` で `**/*.test.*` `**/*.spec.*` `**/tests/**` を検出 |

### 3. エージェント検出

利用可能なエージェントを動的に検出する。詳細は `references/delegation.md` を Read。

### 4. 調査が必要か判断する

調査は**内部（コードベース）調査**と**外部（Web）調査**で担当を分ける。

| 状況 | 判断 |
|---|---|
| 既存コードベースの理解が必要 | **調査担当に委任**（内部調査） |
| 既存実装・依存関係・コード内の技術的制約の確認が必要 | **調査担当に委任**（内部調査） |
| 外部 API / ライブラリの最新仕様・比較・技術トレンドの調査が必要 | **メインセッションが `deep-research` スキルで実施**（調査担当には委任しない） |
| 要件が明確で着手可能 | フェーズ 2 へ進む |

> **なぜ外部調査を委任しないか**: 調査担当（researcher）はコードベース内部調査専任で Web ツールを持たない。`deep-research` は並列サブエージェント（`Task`）を前提とするため、サブエージェントからは完遂できない。外部・Web 調査はメインセッション（オーケストレーター自身）が `Skill` ツールで `deep-research` を直接起動して行う。

### 5. 調査の実施

**内部調査** — Task tool で調査担当エージェントに委任する。テンプレート:

```
以下についてコードベースを調査し、構造化された報告をしてください:

**調査目的**: [何を明らかにしたいか]
**調査対象**: [リポジトリ内のどの領域か]
**具体的に確認してほしい点**:
- [確認事項 1]
- [確認事項 2]

発見事項・推奨事項・リスクをまとめてください。
```

調査担当が見つからなかった場合は、自分で `Glob` / `Grep` / `Read` を使って調査する。

**外部調査** — メインセッション（あなた自身）が `deep-research` スキルを直接起動して行う。調査担当には委任しない。

### 6. プレモーテム（失敗の事前想定）

計画を立てる前に、失敗シナリオを先に考える:

> 「この実装が失敗するとしたら、何が原因か？」

以下の 3 観点から列挙する:

- **技術的リスク** — 依存ライブラリの非互換、パフォーマンス問題等
- **設計リスク** — 既存コードとの不整合、スケーラビリティ
- **テストリスク** — テスト困難な箇所、外部依存

ここで特定したリスクは、フェーズ 2 のテスト戦略の重点や、フェーズ 4 のレビュー時の注目ポイントとして計画に反映する。

### 7. 戦略サマリーの作成

ユーザーに提示する前に、以下を整理する:

- **実装アプローチ** — どう作るか
- **使用するフレームワーク / ライブラリ** — プロジェクト検出の結果を反映
- **リスク・制約事項** — プレモーテムの結果を含む
- **委任先エージェントの確定** — エージェント検出の結果をロールごとに割り当て

戦略サマリーが整ったらフェーズ 2（計画作成）に進む。
