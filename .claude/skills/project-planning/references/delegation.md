# エージェント検出と委任パターン

専門エージェントの動的検出・ロール分類・委任テンプレート・並列/直列の判断をまとめる。

## エージェント検出

**特定のエージェント名をハードコードで前提としない。**
プロジェクトに展開されているエージェントを毎回スキャンしてロール分類する。

### 検出手順

1. `Glob` で エージェント定義ファイルを検出する:

   ```
   Glob: .claude/agents/*.md
   ```

   本リポジトリ開発時は `.apm/agents/*.agent.md` も併用する。

2. 各 `.md` ファイルの**フロントマター**（YAML 部分）を `Read` で読み取り、以下を抽出する:

   - `name` — エージェント名（Task tool での呼び出しに使用）
   - `description` — 能力の説明（ロール分類に使用）
   - `tools` — 利用可能なツール（委任可能な作業範囲の判断に使用）

## ロール分類

description のキーワードに基づき以下のロールに分類する。

| ロール | 分類基準（description のキーワード例） | 用途 |
|---|---|---|
| **テスト担当** | TDD、テスト駆動、テスト品質、Red-Green | テスト作成・TDD プロセスの管理（RED フェーズ） |
| **実装担当** | コード実装支援、開発支援、フレームワーク固有の実装（react / vue / laravel / hono 等のソースコード作成） | フレームワーク固有の**コード**実装（GREEN / IMPROVE フェーズ） |
| **レビュー担当** | コードレビュー、レビュー、品質 | コードレビュー（レビューフェーズ） |
| **セキュリティ担当** | セキュリティ、脆弱性、OWASP | セキュリティレビュー（レビューフェーズ） |
| **調査担当** | コードベース調査、既存実装、依存関係、内部調査 | コードベース**内部**の事前調査（戦略策定フェーズ）。外部・Web 調査は担当外（メインセッションが `deep-research` スキルで実施） |
| **ドキュメント担当** | ドキュメント、文書、Docusaurus、`*-writer` | ドキュメント作成・更新 |

> **分類の優先順位**: description に「ドキュメント」「Docusaurus」が含まれる、または name が `*-writer` のエージェントは、たとえ「実装」「設計」の語を含んでいても **ドキュメント担当**に分類する（コードの実装担当ではない）。例: `docusaurus-writer` は「ドキュメント実装担当」なので実装担当ロールには入れない。

> **NSTG 標準パッケージには実装系・テスト系の専任エージェントは含まれない。** `<framework>-expert` や `tdd-guide` は廃止され、実装・TDD は対応する patterns / tdd-workflow スキルで担う設計になった。したがって「テスト担当」「実装担当」は通常検出されないが、**オーケストレーター（メインセッション）は自分でプロダクションコードを書かない**。この場合はハーネス標準の汎用エージェント `general-purpose` に該当スキルを読ませて委任するのが既定動作になる。利用先プロジェクトが独自に実装系エージェントを追加している場合のみ、下表に従ってそのエージェントに委任する。

## 実装担当エージェントの選択

実装担当が複数検出された場合、プロジェクト検出の結果と照合して最適なものを選ぶ:

1. 検出した実装担当エージェントの `description` / `compatibility` を確認
2. プロジェクトの技術スタック（`package.json` / `composer.json` の内容）と照合
3. 最もマッチするエージェントを選択
   - マッチするエージェントがない場合 → 汎用の `general-purpose` エージェントに該当 `<framework>-patterns` スキルを読ませて委任する（オーケストレーターは自分でコードを書かない）
   - 複数マッチする場合（モノレポ等）→ 対象ファイルに応じて使い分け

## 検出結果の記録

以下の形式で整理して保持し、以降のフェーズで参照する:

```markdown
## 利用可能エージェント

| ロール | エージェント名 | 説明 |
|---|---|---|
| テスト担当 | (検出名) | (description 抜粋) |
| 実装担当 | (検出名) | (description 抜粋) |
| レビュー担当 | (検出名) | (description 抜粋) |
| セキュリティ担当 | (検出名) | (description 抜粋) |
| 調査担当 | (検出名) | (description 抜粋) |
| ドキュメント担当 | (検出名) | (description 抜粋) |
```

## ロールが空の場合の対処

| 状況 | 対処 |
|---|---|
| テスト担当が見つからない | 汎用の `general-purpose` エージェントに `tdd-workflow` スキルを読ませて委任する（実装担当エージェントが居ればそこへ委任してもよい） |
| 実装担当が見つからない | 汎用の `general-purpose` エージェントに該当 `<framework>-patterns` スキルを読ませて委任する。オーケストレーター（メインセッション）は自分でプロダクションコードを書かない |
| レビュー担当が見つからない | レビューフェーズをスキップし、ユーザーに手動レビューを依頼する |
| セキュリティ担当が見つからない | セキュリティレビューをスキップし、レビュー担当に可能な範囲でセキュリティ観点も依頼する |
| 調査担当が見つからない | 自分で `Glob` / `Grep` / `Read` を使い調査する |
| ドキュメント担当が見つからない | ドキュメント変更は自分で実施するか、ユーザーに別タスク化を提案する |

## 委任時の指示テンプレート

Task tool でエージェントに委任する際、以下の情報を必ず含める:

1. **コンテキスト** — プロジェクトの技術スタック、現在のフェーズ、これまでの作業状況
2. **タスク** — 具体的に何をしてほしいか
3. **制約** — 守るべきルール・規約（既存パターンの踏襲、テストランナー等）
4. **成果物** — 何を報告してほしいか（ファイルパス、テスト結果、カバレッジ等）

### 並列委任と直列委任

| パターン | 使用場面 |
|---|---|
| **直列**（順番に委任） | テスト → 実装 → レビュー（前の結果に依存する） |
| **並列**（同時に委任） | コードレビュー + セキュリティレビュー（互いに独立）、複数モジュールの独立した実装 |

並列委任する場合は、1 つの応答内で複数の Task tool 呼び出しをまとめて発行する。

## エージェントからの結果の解釈

| ロール | 確認する結果 |
|---|---|
| テスト担当 | RED / GREEN / IMPROVE の確認結果、カバレッジ数値 |
| 実装担当 | 実装内容、テスト結果、作成 / 変更されたファイル |
| レビュー担当 | 🔴 / 🟡 / 💭 の件数と内容 |
| セキュリティ担当 | 🔴 / 🟡 / 💭 の件数と内容 |
| 調査担当 | 発見事項・推奨事項・リスク |
| ドキュメント担当 | 作成 / 更新されたドキュメントのパス、レビュー要否 |

## 委任の禁則

- **Subagent から Subagent を spawn しない** — 公式制約。委任先エージェントには「他のエージェントに再委任しない」と明示する
- **コードベースの複数領域に渡る委任は分割する** — 1 つのエージェントに大量のスコープを渡さず、ステップごとに細かく委任する
- **自分が書ける範囲のコードは委任しない** — `.plan/` のメモ、コミットメッセージの下書き、PR テンプレートの記入などは自分で実施する
