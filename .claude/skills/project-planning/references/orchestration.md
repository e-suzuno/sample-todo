# 計画作成・TDD 実行・イテレーション管理

フェーズ 2（計画作成）からフェーズ 5（イテレーション）までの詳細手順。

## フェーズ 2: 計画作成

詳細な計画を作成し、**ゲート①**でユーザーの承認を得る。

### 計画フォーマット

以下の形式で計画を提示する:

```markdown
## 実装計画

### 概要
[タスクの要約と目標]

### 利用エージェント
| ロール | エージェント名 |
|---|---|
| テスト担当 | [検出名] |
| 実装担当 | [検出名] |
| レビュー担当 | [検出名] |
| ... | ... |

### 実装ステップ

#### ステップ 1: [名前]
- **種別**: テスト / 実装 / リファクタリング
- **担当ロール**: [テスト担当 / 実装担当 / ...]
- **内容**: [具体的な作業内容]
- **成果物**: [作成 / 変更されるファイル]

#### ステップ 2: [名前]
...

### テスト戦略
- ユニットテスト: [対象と方針]
- インテグレーションテスト: [対象と方針]
- E2E テスト: [必要な場合]

### 非目標（スコープ外）
このタスクで**対応しない**ことを明示する:
- [やらないこと 1]: [理由 / 別タスクとして切り出す場合はその旨]
- [やらないこと 2]: [理由]

### リスク評価
| リスク | 発生可能性 | 影響度 | 軽減策 |
|---|---|---|---|
| [リスク 1] | 高/中/低 | 高/中/低 | [対策] |
| [リスク 2] | 高/中/低 | 高/中/低 | [対策] |

### ロールバック基準
問題が発生した場合の撤退基準:
- [条件]: [リバート / 修正 / ユーザー報告]
```

### 承認ゲート ①

計画を提示し、ユーザーの承認を求める:

- 「この計画で進めてよいですか？」
- ユーザーが修正を求めた場合は計画を更新して再提示する
- **明示的な承認があるまでフェーズ 3 に進まない**

## フェーズ 3: セットアップ

`git-setup` スキルの手順に従って以下を実行する:

1. Git 環境の検出（GitHub 有無・デフォルトブランチ・`gh` CLI）
2. フィーチャーブランチの作成
3. GitHub Issue の起票（または `.plan/` に計画ファイル出力）

Issue 番号を取得できた場合は、以降のコミットメッセージ・PR タイトルに使用する。

## フェーズ 4: 実行（TDD サイクル）

計画の各実装ステップについて、以下のサイクルを回す。

> **テスト担当・実装担当エージェントが検出されない場合（NSTG 標準パッケージの既定）でも、メインセッションは自分でコードを書かない。** ハーネス標準の汎用エージェント `general-purpose` に `tdd-workflow` と該当 `<framework>-patterns` スキルを読ませて RED / GREEN / IMPROVE を委任する。以下の「〜に委任」手順は、専任の実装系・テスト系エージェントを検出できた場合はそのエージェントに、検出できない場合は `general-purpose` に対して適用する。

### 4-1. テスト作成（RED）— テスト担当に委任

Task tool でテスト担当エージェントに以下を委任する:

```
以下の要件に対するテストを作成してください（RED フェーズ）:

**要件**: [ステップの要件]
**対象**: [テスト対象のモジュール / コンポーネント / エンドポイント]
**テスト種別**: [ユニット / インテグレーション / E2E]
**プロジェクト情報**: [フレームワーク・テストランナー・既存テストの場所]

以下を行ってください:
1. テストケースを設計する（正常系 + エッジケース）
2. テストを実装する
3. テストを実行し、失敗（RED）を確認する
4. 結果を報告する（テストファイルパス・失敗内容）
```

### 4-2. 実装（GREEN）— 実装担当に委任

```
以下のテストを通す最小限の実装を行ってください（GREEN フェーズ）:

**テストファイル**: [テストファイルパス]
**要件**: [実装要件]
**プロジェクト情報**: [フレームワーク・既存コードの構造]

以下を行ってください:
1. テストを通す最小限のコードを実装する
2. テストを実行して GREEN を確認する
3. 結果を報告する（実装ファイルパス・テスト結果・カバレッジ）
```

### 4-3. リファクタリング（IMPROVE）— 実装担当に委任

```
テストがグリーンを維持したまま、以下のリファクタリングを行ってください（IMPROVE フェーズ）:

**対象ファイル**: [ファイルパス一覧]
**改善ポイント**: [重複除去 / 命名改善 / 構造整理 等]

以下を行ってください:
1. コードの品質を改善する
2. テストを実行してグリーンを維持していることを確認する
3. 結果を報告する（変更内容・テスト結果）
```

### 4-4. レビュー — レビュー担当に委任

#### コードレビュー（必須）

```
以下の変更をレビューしてください:

**対象**: 現在のブランチの変更全体（git diff で確認できる変更）
**コンテキスト**: [何を実装したか・目的]

🔴🟡💭 の優先度マーカー付きで報告してください。
```

#### セキュリティレビュー（認証 / 認可 / データ処理を含む変更の場合）

セキュリティ担当エージェントが検出されている場合、Task tool で委任する:

```
以下の変更のセキュリティレビューを行ってください:

**対象**: 現在のブランチの変更全体（git diff で確認できる変更）
**コンテキスト**: [何を実装したか・扱うデータの種類]

🔴🟡💭 の優先度マーカー付きで報告してください。
```

**コードレビューとセキュリティレビューは独立しているため、並列で委任する。**

## フェーズ 5: イテレーション

レビュー結果に基づき、品質基準を満たすまでサイクルを繰り返す。

### 判定ロジック

```
レビュー結果を受領
  ├── 🔴 ブロッカーあり → 修正サイクルへ
  │     ├── テスト修正が必要 → テスト担当に委任（4-1 へ戻る）
  │     ├── 実装修正が必要 → 実装担当に委任（4-2 へ戻る）
  │     └── セキュリティ修正が必要 → 実装担当に委任（4-2 へ戻る）
  │     → 修正後、再レビュー（4-4 へ戻る）
  │
  ├── 🟡 提案のみ（🔴 なし）→ 対応判断
  │     ├── 軽微な改善 → 実装担当に委任して対応
  │     └── 大規模なリファクタリング → 次回対応として記録
  │
  └── 🔴 なし → フェーズ 6 へ
```

### イテレーション上限

- 最大 **3 回** のイテレーションを目安とする
- 3 回を超えてもブロッカーが解消しない場合、ユーザーに状況を報告し判断を仰ぐ:
  - 残存ブロッカー一覧
  - 試行した修正内容
  - 推定原因と対策案

## エラーハンドリングとリカバリー

### テスト失敗からのリカバリー

```
テストが失敗（RED 解消できない）
  ├── 環境の問題（依存関係・設定） → 自分で Bash で調査・修正
  ├── テスト自体の問題（誤ったアサーション） → テスト担当に修正を委任
  └── 実装の根本的な問題 → 計画を見直し、ユーザーに相談
```

### レビューブロッカーからのリカバリー

```
🔴 ブロッカーの対処
  ├── ロジックバグ → 実装担当に修正を委任
  ├── セキュリティ脆弱性 → 実装担当に修正指示（セキュリティ担当の指摘を添付）
  └── 設計上の根本問題 → 計画を見直し、ユーザーに相談
```

### イテレーション上限到達

3 回のイテレーションで解決しない場合:

1. 残存問題の一覧を作成する
2. 試行した修正内容を記録する
3. ユーザーに以下を提示する:
   - 現状のサマリー
   - 解決案の提案
   - 手動介入が必要な箇所
