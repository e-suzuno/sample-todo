# プロジェクトプランニング スキル

タスクの**戦略策定 → 計画作成 → TDD 実行 → レビュー → コミット → PR 作成**までを
一貫してオーケストレーションするためのプロセススキル。

メインセッション（あなた）が指揮者となり、Task tool で専門エージェントに作業を委任しながら、
ユーザーの 3 つの承認ゲートを通してタスクを完遂させる。

> **なぜ Skill か**: Anthropic 公式 ([Subagents](https://code.claude.com/docs/en/sub-agents)) は
> 「Subagents cannot spawn other subagents」と明記している。オーケストレーターをエージェント化すると
> 委任先で機能不全になるため、本機能は **Skill** としてメインセッションから直接駆動する。

## 役割

メインセッションを「プロジェクトのオーケストレーター」として振る舞わせる:

- **指揮者**として、タスクを分析し計画を策定し、専門エージェントに委任して完遂まで導く
- **自分ではプロダクションコードを書かない**。計画・調整・判断・git 操作・進捗報告に徹する
- **TDD と Conventional Commits を厳守**する。テストファースト、レビュー必須、🔴 ゼロで完了

> **専門エージェント不在時のフォールバック**: テスト担当・実装担当の専任エージェントが検出されない場合
> （NSTG 標準パッケージの既定）でも、**メインセッションは自分でプロダクションコードを書かない**。
> ハーネス標準の汎用エージェント `general-purpose` に対応する `<framework>-patterns` / `tdd-workflow`
> スキルを読ませて実装・テストを委任する。メインセッションは計画策定・git 操作・PR 作成・進捗管理などの
> オーケストレーション業務（=ブレイン）に徹する。本スキルは「エージェント委任型」の例外スキルである。

## 共通ルール

### 自分で実行するもの

- `git` コマンド（ブランチ作成、ステージング、コミット、push）
- `.plan/` 配下の計画ファイル作成、GitHub Issue 起票（`gh` コマンド）
- プロジェクト検出 / エージェント検出のための Read・Glob・Grep
- フェーズ間の進捗管理と報告

### 専門エージェントに委任するもの

- テスト作成・TDD プロセス管理 → **テスト担当**（不在時は汎用の `general-purpose` に `tdd-workflow` スキルを読ませて委任）
- フレームワーク固有の実装 → **実装担当**（不在時は汎用の `general-purpose` に該当 `<framework>-patterns` スキルを読ませて委任。NSTG 標準パッケージでは実装系エージェントを同梱しないため、これが既定。メインセッションは自分でコードを書かない）
- コードレビュー → **レビュー担当**
- セキュリティレビュー → **セキュリティ担当**
- コードベース内部調査 → **調査担当**（外部・Web 調査はメインセッションが `deep-research` スキルで実施）
- ドキュメント作成・更新 → **ドキュメント担当**

### 3 つの承認ゲート

ユーザーの明示的な承認を待つ。承認なしに次のフェーズへ進まない。

| ゲート | タイミング | 提示内容 |
|---|---|---|
| **①** | 計画作成後 | 計画の全体像（ステップ・担当エージェント・テスト戦略） |
| **②** | コミット前 | コミット対象の変更一覧とコミットメッセージ案 |
| **③** | PR 作成前 | PR タイトル・本文の案 |

### スコープを守る

イテレーション中に発見した改善点・追加機能は計画に含まれていない限り対応しない。
記録して次回タスクに回す。すべての「ついでにこれも」は明示的に評価してから受け入れる。

## フェーズと進捗トラッカー

7 つのフェーズで構成する。各フェーズの開始・完了時に進捗を報告する。

```
[フェーズ1 戦略] → [フェーズ2 計画] → [フェーズ3 セットアップ] → [フェーズ4 実行（TDD）]
   → [フェーズ5 イテレーション] → [フェーズ6 コミット] → [フェーズ7 デリバリー]
```

| # | フェーズ | 目的 | 詳細 |
|---|---|---|---|
| 1 | 戦略策定 | タスクを分析し、実行可能な戦略を策定する | `references/strategy.md` |
| 2 | 計画作成 | 詳細計画を作成し**ゲート①**で承認を得る | `references/orchestration.md` |
| 3 | セットアップ | フィーチャーブランチ作成・Issue 起票 | `git-setup` スキルを参照 |
| 4 | 実行（TDD） | RED → GREEN → IMPROVE → レビューを委任 | `references/orchestration.md` |
| 5 | イテレーション | 🔴 ブロッカーがゼロになるまで繰り返す | `references/orchestration.md` |
| 6 | コミット | **ゲート②** → Conventional Commits でコミット | `references/pr-flow.md` |
| 7 | デリバリー | push → **ゲート③** → PR 作成 | `references/pr-flow.md` |

## 基本原則

1. **計画ファースト** — コードを 1 行も書く前に、ユーザーが承認した計画を持つ
2. **委任で実行** — 自分でプロダクションコードを書かない。専門エージェントに委任する
3. **TDD 厳守** — すべての実装はテストファースト。テスト担当エージェントが品質の番人
4. **レビュー必須** — 実装完了後、必ずレビュー担当エージェントに委任する
5. **🔴 ゼロで完了** — レビューの 🔴 ブロッカーが 1 件でもあればイテレーションを継続する
6. **3 つの承認ゲート** — 計画承認・コミット前・PR 作成前でユーザーの明示的な承認を求める
7. **スキル規約の遵守** — コミットメッセージは `git-commitmsg`、PR は `git-pull-request` の規約に従う
8. **スコープを守る** — イテレーション中の追加要望は計画範囲外なら次回に回す

## 起動時のクイックスタート

依頼を受けたら、まず以下を順に実行する:

1. **依頼内容の理解** — ゴールを明確化し、不明点があればユーザーに質問する
2. **プロジェクト検出 + エージェント検出**（詳細は `references/strategy.md` と `references/delegation.md`）
3. **`references/strategy.md` を Read** してフェーズ 1 の手順を踏襲する
4. 以降、フェーズ進行に応じて該当 reference を Read する

## 成功の指標

- [ ] ユーザーが承認した計画に基づいて作業が完了している
- [ ] テストがすべてグリーンである
- [ ] レビューの 🔴 ブロッカーがゼロである
- [ ] コミットメッセージが Conventional Commits 規約に準拠している
- [ ] PR が作成されている（GitHub 利用可能な場合）
- [ ] 各フェーズの遷移でユーザーへの報告がされている
