# ADR（Architecture Decision Record）フォーマット定義

対象: ドキュメントルート配下の `adr/` ディレクトリ内のファイル

## ファイル命名規則

```
adr-<3桁番号>-<英語スラッグ>.md
例: adr-008-cognito-user-pool.md
```

次の番号は ADR ディレクトリ内の `index.md` の一覧を確認して採番する。

## テンプレート

```markdown
---
sidebar_position: <番号+1>
title: "ADR-<番号>: <タイトル>"
---

# ADR-<番号>: <タイトル>

- **ステータス**: 承認済み
- **日付**: YYYY-MM-DD
- **関連ドキュメント**: [ドキュメント名](相対パス)、[ドキュメント名](相対パス)

## コンテキスト

[なぜこの決定が必要になったか。背景・制約・要件を記述する。]
[箇条書きで制約や課題を列挙すると読みやすい。]

## 決定

[何を決めたか。採用した選択肢を1文で明確に書く。]
[必要に応じて箇条書きで補足する。]

## 理由

[なぜその選択肢を選んだか。比較表を含めるのが望ましい。]

| 方式 | メリット | デメリット |
|------|---------|-----------|
| **採用した方式（採用）** | ... | ... |
| 代替案1 | ... | ... |
| 代替案2 | ... | ... |

## 結果

[この決定によって生じる具体的な影響・制約・設定事項を箇条書きで記載する。]
[将来の開発者が知っておくべき注意点も含める。]
```

## 各セクションの書き方

### コンテキスト
- 「〜という課題があった」「〜という制約があった」と具体的に書く
- 技術的な背景だけでなく、ビジネス・運用上の理由も書く
- 具体的な数値（サイズ上限、レート制限等）があれば含める

### 決定
- 「〇〇を採用する」と明確に1文で書く
- 補足が必要な場合は箇条書きで構成要素を説明する

### 理由
- 採用しなかった選択肢との比較表を入れると説得力が増す
- テーブル形式で「方式 / メリット / デメリット」を対比する
- 採用した方式は **太字** で強調する

### 結果
- この決定によって生じる具体的な影響を箇条書きで記載する
- 設定値、制約、注意点など実装者が知っておくべき情報を含める
- 良い面だけでなく許容するトレードオフ（遅延の許容等）も正直に書く

## 作成後の更新

ADR ディレクトリ内の `index.md` の一覧テーブルに新しいADRの行を追加する。

## ステータスの種類

| ステータス | 意味 |
|---|---|
| `提案中` | 議論中・レビュー待ち |
| `承認済み` | 採用・実施済み |
| `廃止` | 廃止（後継あり） |
| `ADR-XXX に置換` | 別のADRに置き換えられた |
