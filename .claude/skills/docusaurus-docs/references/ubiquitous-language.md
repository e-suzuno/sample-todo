# ユビキタス言語 フォーマット定義

対象: ドキュメントルート配下の `ubiquitous-language.md`
- ブートストラップで合意済みの場合はそのパス（既定は `basic-design/ubiquitous-language.md`）
- ファイルが存在せず、かつディレクトリ構成の合意もされていない場合は、先に `references/bootstrap.md` を実行すること

## 重要性

このファイルはプロジェクト全体の命名基準となる。
コード・API・ドキュメント間で命名が一致するよう、このファイルが唯一の正とする。

## フォーマット

冒頭のフロントマターは必須（採番は SKILL.md「フロントマター採番ルール」に従う）。

|---|---|---|---|

## 各セクションの書き方

### ドメイン用語テーブル
- 用語（日本語）: プロジェクト内で使う日本語の概念名
- 英語キー: コード上の識別子（変数名・フィールド名・APIパスに使う形）
- 定義: 「何か」を1文で書く。略語・省略形は使わない

### フィールド命名規則
- 統一名: API・DB・UI で共通して使うフィールド名
- 非推奨: 同じ概念に対して使ってはいけない別名を列挙する（混在を防ぐため）
- 使用箇所: そのフィールドが現れる場所（API, DB, UI 等）

### 日本語表記ルール
- UI表示やドキュメントで揺れやすい日本語表記の統一ルールを記載する
- 例: 「ログイン」と「サインイン」のどちらを使うか

### ステータス値
- システム内で使用する列挙値を定義する
- 値（英語）、日本語表記、説明の3カラムで記載する
- ドメインごとにサブセクション（`###`）で分けてよい

### 廃止用語
- 用語を変更した場合、古い名称を削除せず廃止として残す
- 移行期間中に混乱を防ぐためのリファレンスになる
- 廃止用語がまだない場合、このセクションは省略してよい

## 命名の原則

- 略語・省略形は避ける（`upload` を `upl` にしない）
- 複数形・単数形は英語として自然な方を採用し、一貫させる
- フィールドのケース規則は既存コードの命名規則に合わせる（camelCase or snake_case）

## 注意事項

用語を**変更・削除**する場合は影響範囲が広いため、
変更前にコード全体を Grep して影響箇所を確認すること。
大規模な命名変更はADRへの記録も検討する。
