# 非機能要件（NFR）フォーマット定義

対象: ドキュメントルート配下の `non-functional-requirements.md` / `nfr.md`
- ブートストラップで合意済みの場合はそのパス（既定は `basic-design/non-functional-requirements.md`）
- ディレクトリ構成が未確立の場合は、先に `references/bootstrap.md` を実行すること

## セクション構成

非機能要件は「カテゴリごとに数値目標と根拠を書く」形式で構成する。
数値が未定の項目は **「未定」と明示**し、決まり次第更新する前提とする（空欄にしない）。

```markdown
---
sidebar_position: <番号>
title: "非機能要件"
---

# 非機能要件

## 概要

[対象システム・対象範囲（全機能 / 特定サブシステム）・合意ステータスを 1〜3 文]

## 可用性

| 項目 | 目標値 | 計測・根拠 |
|---|---|---|
| 稼働率 | 99.9%（月次） | 月間ダウンタイム 43 分以内を許容 |
| 計画停止 | 日曜 2:00〜4:00 | 計画停止は稼働率から除外 |
| RTO（目標復旧時間） | 1 時間 | 障害発生から本番復旧まで |
| RPO（目標復旧地点） | 15 分 | データ損失の許容範囲 |

## 性能

| 項目 | 目標値 | 計測・根拠 |
|---|---|---|
| 画面表示レイテンシ（P95） | 1 秒以内 | 一覧系画面、主要なデータフェッチ含む |
| API レスポンスタイム（P95） | 500 ms 以内 | 読み取り系 API |
| API レスポンスタイム（P95） | 1 秒以内 | 書き込み系 API |
| 同時アクセスユーザー数 | 500 人 | ピーク時想定 |

## スケーラビリティ

- [水平スケールの方針 / 垂直スケールの方針]
- [将来の成長見込み（例: 年 2 倍のデータ増加に耐える設計）]
- [スケール限界と、それに達した場合の次アクション]

## セキュリティ

| 項目 | 要件 |
|---|---|
| 通信暗号化 | TLS 1.2 以上 |
| 認証 | MFA 必須（管理者）/ 任意（一般ユーザー） |
| 認可 | ロールベースアクセス制御（RBAC） |
| 監査ログ | 認証・権限変更・機密データアクセスを記録 |
| 個人情報保護 | 個人情報保護法・GDPR 準拠（該当する場合） |

## 運用性

- [監視対象（インフラ / アプリケーション / ビジネス指標）]
- [アラート方針（P1 / P2 / P3 の定義と通知先）]
- [ログ保持期間と保管場所]
- [バックアップ方針（頻度 / 保持期間 / 復元テスト）]

## 互換性

- [対応ブラウザと最小バージョン]
- [対応モバイル OS]
- [API の後方互換性方針（バージョニング戦略）]

## 保守性

- [リリース頻度の目標]
- [テストカバレッジの目標（単体 / 結合 / E2E）]
- [技術的負債への対応方針]
```

## 書き方の指針

- **具体的な数値目標を書く** — 「高速」「高可用」等の曖昧な表現は避ける
- **未確定は「未定」と明記する** — 空欄や記載漏れは要件漏れと区別できないため、明示的に「未定」と書いて根拠も添える（例: 「未定 — 性能試験実施後に確定予定」）
- **計測方法・根拠を併記する** — 数値だけでは測れない。どうやって計測するか、なぜその値かを示す
- **RTO / RPO / P95 等の用語を使う場合は、ドキュメント冒頭かユビキタス言語定義で定義を揃える**
- **SLA / SLO / SLI の区別を意識する** — 対外約束（SLA）／内部目標（SLO）／計測指標（SLI）を混同しない

## カテゴリの選定

プロジェクトの性質に応じて節を追加・削除してよい。基本カテゴリ:

- **可用性** — 稼働率・RTO / RPO
- **性能** — レイテンシ・スループット・同時接続
- **スケーラビリティ** — 成長見込みとスケール戦略
- **セキュリティ** — 認証・認可・暗号化・監査
- **運用性** — 監視・アラート・ログ・バックアップ
- **互換性** — ブラウザ・OS・API バージョニング
- **保守性** — リリース頻度・テスト方針

B2C Web アプリなら性能・可用性重視、社内ツールなら保守性・運用性重視、といった具合にメリハリを付ける。

## 書いてはいけないもの

- 具体的な実装技術名（「Redis を使う」「nginx で負荷分散する」等）— これは ADR で記録する
- インフラ構成図の詳細 — 別途 `architecture.md` 等で管理
- 監視ダッシュボードの URL・ツール固有の設定値 — 運用ドキュメントで管理
- テストケース一覧 — テスト仕様書で管理

## 作成後の更新

- 要件が変更されたら**変更履歴**節を追加し、変更日・変更内容・承認者を記録する
- 「未定」項目が確定したら値を更新し、変更履歴に記録する
- 目標値を大きく変更する場合は ADR で判断経緯を記録する
