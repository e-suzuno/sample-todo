# エラーコード一覧

## 概要

本システムで返却するエラーコードの一覧。API レスポンス・画面表示・ログ出力での統一的な参照キーとなる。
具体的なメッセージ文言はコードで管理し、本ドキュメントは「コード / HTTP ステータス / 意味 / 発生条件 / 対処」の対応を定める。

## 命名規則

- 形式: `<DOMAIN>_<CONDITION>` （SCREAMING_SNAKE_CASE）
- 例: `AUTH_UNAUTHORIZED`、`USER_NOT_FOUND`、`ORDER_ALREADY_CANCELED`
- ドメイン略称はユビキタス言語定義に準拠する

## 共通エラー（認証・権限・リクエスト）

| コード | HTTP | 意味 | 発生条件 | 対処（クライアント側） |
|---|---|---|---|---|
| `AUTH_UNAUTHORIZED` | 401 | 認証失敗 | トークン未提示・期限切れ・改ざん | 再ログインへ誘導 |
| `AUTH_FORBIDDEN` | 403 | 権限不足 | 認証済みだが操作権限が無い | 権限を持つユーザーに依頼する旨を表示 |
| `REQUEST_INVALID` | 400 | リクエスト不正 | バリデーションエラー・必須項目不足 | 入力フォームにフィールド別エラー表示 |
| `REQUEST_TOO_LARGE` | 413 | リクエストサイズ超過 | アップロードサイズ上限超過 | 上限値を明示し再試行を促す |
| `RATE_LIMIT_EXCEEDED` | 429 | レート制限超過 | 短時間の過剰リクエスト | リトライまでの待機時間を案内 |

## ドメイン別エラー

### ユーザー関連

| コード | HTTP | 意味 | 発生条件 | 対処 |
|---|---|---|---|---|
| `USER_NOT_FOUND` | 404 | ユーザー未存在 | 指定 ID のユーザーが存在しない | 一覧に戻す |
| `USER_EMAIL_DUPLICATED` | 409 | メール重複 | 既に登録済みのメールで新規登録試行 | 別メール入力を促す |
| `USER_SUSPENDED` | 403 | アカウント停止中 | `status = suspended` のユーザーが認証試行 | サポート連絡先を表示 |

## システムエラー

| コード | HTTP | 意味 | 発生条件 | 対処 |
|---|---|---|---|---|
| `INTERNAL_ERROR` | 500 | サーバー内部エラー | 予期しない例外 | 汎用エラー画面へ、エラー ID を表示 |
| `SERVICE_UNAVAILABLE` | 503 | 一時的に利用不可 | 計画停止・依存サービス障害 | 復旧予定時刻を表示（わかる場合） |
| `TIMEOUT` | 504 | タイムアウト | 外部 API / DB 応答なし | 再試行を促す |

## 書き方の指針

- **コード名を正とする** — API レスポンス・画面表示・ログ出力・テストコードすべてでこのコード名を参照する前提で運用する
- **メッセージ文言は書かない** — 画面表示文言は i18n リソース / コードで管理。ここではあくまで「意味」を書く
- **「対処」は利用者（ユーザー or API クライアント）の次アクションを書く** — 「システムが何をすべきか」ではない
- **HTTP ステータスは必ず対応付ける** — 同じコードで複数のステータスを返さない
- **発生条件は具体的に書く** — 「エラー発生時」のような抽象表現は避ける

## カテゴリ分割の方針

エラー数が増えたらドメイン別に節を分ける。目安は 15 件を超えたら分割検討:

- **共通エラー** — 認証・権限・リクエスト形式（どのドメインでも発生しうる）
- **ドメイン別エラー** — ユーザー・注文・支払い・商品等、ドメインごとに節を分ける
- **システムエラー** — 500 系・外部サービス障害・タイムアウト等

## HTTP ステータスコード選定の基本

| ステータス | 用途 |
|---|---|
| 400 | リクエスト形式不正（バリデーションエラー含む） |
| 401 | 認証失敗（誰であるか証明できない） |
| 403 | 権限不足（認証済みだが操作不可） |
| 404 | リソース未存在 |
| 409 | 競合（重複・状態遷移不可） |
| 422 | 意味的に処理不能（ビジネスルール違反） |
| 429 | レート制限超過 |
| 500 | サーバー内部エラー |
| 503 | 一時的な利用不可 |
| 504 | タイムアウト |

401 と 403 の使い分け・400 と 422 の使い分けは最初に方針を決めて全エラーに一貫適用する。

## OpenAPI との整合

- 各エンドポイントの `responses` セクションでは、返却しうるエラーコードを列挙する
- エラーレスポンスのスキーマは共通化する（例: `ErrorResponse` スキーマを `components/schemas/` に定義し、`code` / `message` / `details` を持たせる）
- 本ドキュメントのコードと OpenAPI の `ErrorResponse.code` に記載される enum は一致させる

## 書いてはいけないもの

- 具体的なエラーメッセージ文言（コード / i18n で管理）
- スタックトレースや実装詳細（ログで管理）
- エラーハンドリングのコード例（実装コードで管理）
- 画面ごとのエラー表示デザイン（画面定義で管理）

## 作成後の更新

- 新規エラーコードを追加したら、以下を並行して更新する:
  1. 本ドキュメントの該当節
  2. OpenAPI 定義の `responses` と `ErrorResponse.code` enum
  3. コード側のエラーコード定義（定数・enum）
  4. エラーコードに対応する画面表示文言リソース
- 廃止コードは削除せず「廃止」マークを付けて履歴を残す（既存クライアントが参照している可能性があるため）
- HTTP ステータスの変更は破壊的変更のため、API バージョニングの判断と紐付ける（ADR で記録）
