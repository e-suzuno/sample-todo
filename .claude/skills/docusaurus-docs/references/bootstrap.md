# ブートストラップ手順（レガシー / 新規プロジェクト向け）

対象: ドキュメント環境がまだ整備されていないプロジェクト。以下のいずれかに該当するときに実行する。

- Docusaurus が未導入（`docusaurus.config.{ts,js}` が存在しない）
- `docs/basic-design/` または `docs/detailed-design/` が存在しない
- `ubiquitous-language.md` が存在しない
- OpenAPI 定義（`openapi.yaml` / `openapi.yml` / `openapi.json`）が存在しない

## 目的

レガシー/新規プロジェクトで発生しがちな以下の事故を防ぐ:

- エージェントが独自ルールでディレクトリ構成を決めてしまう
- ユビキタス言語の欠落した状態で画面定義・シーケンス図を書き始める
- OpenAPI 未定義のまま API 仕様を画面定義とシーケンス図に重複記載する
- 既存の Excel/Word 設計書や旧 Wiki を無視してゼロから書き始める

**基本設計 / 詳細設計** というカテゴリ前提を最初にユーザーと合意した上で、後続のドキュメント作成を安定させる。

## フェーズ全体像

```
Phase 0  Docusaurus 導入確認
   ↓
Phase A-0  既存ドキュメント資産のヒアリング
   ↓
Phase A-1  ディレクトリ構成の合意
   ↓         ┌─ Phase A-2  ユビキタス言語  ┐
   ├───────→ │                            │ 並列実行可
   └─────┐   └─ Phase A-3  OpenAPI 雛形   ┘
             ↓
Phase B  推奨セット（AskUserQuestion で選択）
   ↓
Phase C  通常フローへ復帰
```

---

## Phase 0 — Docusaurus 導入状況の確認

ドキュメントルート特定の前に、**Docusaurus 自体が導入されているか**を確認する。

### 判定手順

1. プロジェクトルートから `docusaurus.config.js` または `docusaurus.config.ts` を Glob で探す
2. 見つかれば Phase A-0 へ進む
3. 見つからない場合は以下の AskUserQuestion で方針を確認する

### AskUserQuestion 例（Docusaurus 未導入時）

```
質問: このプロジェクトに Docusaurus を導入してドキュメント環境を構築してよいですか？
- 導入する（推奨） — Docusaurus を新規セットアップしてドキュメント環境を立ち上げる
- 既存の別ドキュメントツールを使う — Docusaurus 前提の本スキルは使えない旨を伝えて終了
- ドキュメントルートのパスだけ決めて Markdown で始める — Docusaurus は後で導入する
```

「導入する」を選んだ場合、以下の **assets のピン留め版**を第一選択として使用する:

- `assets/docusaurus-package-example.json` — 動作実績のあるバージョン組み合わせ（**これを優先**）
- `assets/docusaurus-config-example.ts` — config 雛形

作成するファイル:

- `package.json`（`docusaurus-package-example.json` をコピーし、`name` をプロジェクト名に置換）
- `docusaurus.config.ts`（`docusaurus-config-example.ts` を参考）
- `sidebars.ts`（`autogenerated` ベース推奨）
- `docs/` ディレクトリ

### 標準搭載プラグイン

`docusaurus-package-example.json` には以下のプラグイン・テーマが**標準で含まれている**。bootstrap 時にユーザーに導入可否を個別確認しない。不要なら後でプロジェクト側が `package.json` / `docusaurus.config.ts` から外す運用とする。

- `docusaurus-plugin-openapi-docs` + `docusaurus-theme-openapi-docs` — OpenAPI 仕様の UI 表示
- `docusaurus-plugin-drawio` — `.drawio` ファイルの埋め込み
- `@docusaurus/theme-mermaid` — Mermaid 図のサポート
- `@easyops-cn/docusaurus-search-local` — ローカル検索

### 導入時のバージョン方針

- **第一選択**: `docusaurus-package-example.json` のバージョンをそのまま採用する
- **独自判断でメジャーバージョンを上げない** — 特に Docusaurus 3 系 → 4 系の自動アップグレードは禁止（プラグイン互換性が未確認のため）
- **互換性のある最新マイナーを使いたい場合のみ**、各パッケージの npm 最新版を確認した上でユーザーに変更提案してから採用する
- **プラグイン間のバージョン同期** — `docusaurus-plugin-openapi-docs` と `docusaurus-theme-openapi-docs` は必ず同一マイナーバージョンで揃える
- **Node.js 要件** — `engines.node >=20.0` を守る

完了条件: `npm install && npm run build` が成功すること。

### サイドバー（sidebars.js）の落とし穴

`sidebars.js` を `autogenerated` で組む際、`{ type: 'autogenerated', dirName: 'basic-design' }` と書くと、**`basic-design/_category_.json` による外側ラップは無視される**（`dirName` で指定したディレクトリの中身が直接展開されるため）。

サイドバーにも「基本設計」という日本語ラベルのカテゴリを表示したい場合は、以下のいずれかを選ぶ。

**選択肢 1: ルート全体を autogenerated にする**（推奨・最もシンプル）

```js
mainSidebar: [{ type: 'autogenerated', dirName: '.' }]
```

こうすると `basic-design/_category_.json` や `detailed-design/_category_.json` が自動認識され、カテゴリ名が反映される。

**選択肢 2: 手動で category ラップする**（サブカテゴリを並列に細かく制御したいとき）

```js
mainSidebar: [
  {
    type: 'category',
    label: '基本設計',
    link: { type: 'generated-index' },
    items: [{ type: 'autogenerated', dirName: 'basic-design' }],
  },
  // 詳細設計も同様に category ラップする
]
```

**やりがちな失敗**: `dirName: 'basic-design'` を sidebar 直下に並べると、ラベルが消える（`_category_.json` の label が適用されない）。bootstrap 直後はまず選択肢 1（`dirName: '.'`）で始めること。

## Phase A-0 — 既存ドキュメント資産のヒアリング

**重要**: ゼロから書き始める前に、必ず既存資産を確認する。レガシープロジェクトでは Excel/Word 設計書、旧 Wiki、Confluence、Redmine 等に大量の情報が埋もれていることが多い。これらを無視すると**同じ情報を別の形式で二重管理**する最悪のパターンに陥る。

### AskUserQuestion 例

```
質問: 既存のドキュメント資産はありますか？（複数選択可、multiSelect: true）
- 既存の Excel/Word 設計書（要件定義書・基本設計書・詳細設計書）
- 旧 Wiki / Confluence / Notion / Redmine
- README.md や docs/ 配下の既存 Markdown
- コード内コメント・JSDoc のみ
- 何も無い（完全新規またはレガシーでドキュメント消失）
```

### ヒアリングの深掘り（資産ありの場合）

選択された資産それぞれについて追加で確認する:

1. **参照可否** — 本スキルからアクセスできるパス/URLか、スクショ等で提供されるか
2. **取り込み方針** — 以下から選択:
   - リライト（内容を読み取り Docusaurus 用に書き直す）
   - 添付（原本を `assets/legacy/` 等に置きリンクのみ）
   - 参照リンクのみ（外部 URL をドキュメントに埋め込む）
   - 無視（古すぎるなど）
3. **優先度** — bootstrap で先に取り込むか、後回しにするか

取り込み方針は `docs/README.md` に「既存ドキュメント資産の扱い」節として記録する。

### 完了チェックリスト

- [ ] 既存資産の棚卸しが AskUserQuestion で完了している
- [ ] 各資産の取り込み方針が `docs/README.md` に記録されている
- [ ] 「無視」と決めた資産についてその理由が記録されている

## Phase A-1 — ディレクトリ構成の合意

ドキュメントルート配下のディレクトリ構成をユーザーに確認する。
**エージェントが勝手に決めない**。必ず以下の AskUserQuestion で合意を取る。

### AskUserQuestion 例

```
質問: ドキュメントのカテゴリ構成をどうしますか？
- basic-design / detailed-design の2カテゴリ（推奨）
  → 基本設計と詳細設計に分ける日本的な設計ドキュメント構成
- 既存の構成に合わせる
  → Phase A-0 で見つかった既存 Markdown の構成を踏襲する
- 別の命名で3カテゴリ以上に分ける
  → requirements / basic-design / detailed-design など
```

### 推奨構成（「basic-design / detailed-design の2カテゴリ」を選んだ場合）

```
<ドキュメントルート>/
├── README.md                          # ドキュメント構成・未整備一覧
├── basic-design/
│   ├── _category_.json
│   └── ubiquitous-language.md
└── detailed-design/
    ├── _category_.json
    ├── api/
    │   └── openapi.yaml
    ├── screens/
    └── sequence.md
```

### Docusaurus 固有の成果物

カテゴリごとに `_category_.json` を配置する。これが無いとサイドバーのラベル・順序が制御できない。

**`basic-design/_category_.json` の例**:

```json
{
  "label": "基本設計",
  "position": 1,
  "link": {
    "type": "generated-index",
    "description": "システムの基本設計ドキュメント（ユビキタス言語・アーキテクチャ・非機能要件等）"
  }
}
```

**`detailed-design/_category_.json` の例**:

```json
{
  "label": "詳細設計",
  "position": 2,
  "link": {
    "type": "generated-index",
    "description": "詳細設計ドキュメント（画面定義・シーケンス図・API仕様・ER図等）"
  }
}
```

ラベルは必ず日本語にする（本スキルの共通ルール「ドキュメントは日本語で記述する」に準拠）。

### 完了チェックリスト

- [ ] カテゴリ構成が AskUserQuestion で合意されている
- [ ] 合意内容が `docs/README.md` の「ドキュメント構成」節に記録されている
- [ ] 各カテゴリに `_category_.json` が配置されている
- [ ] `npm run build` が成功し、サイドバーに合意どおりのカテゴリが日本語ラベルで表示される

## Phase A-2 & A-3 — 並列実行可

Phase A-1 の合意が取れた後、**A-2（ユビキタス言語）と A-3（OpenAPI）は並列で実行してよい**。
両者に依存関係は無い（ユビキタス言語のフィールド名が OpenAPI に反映される方が望ましいが、bootstrap 段階では start-small で始めるためどちらが先でも構わない）。

### Phase A-2 — ユビキタス言語の起票

`basic-design/ubiquitous-language.md`（A-1 で合意したパス）を `references/ubiquitous-language.md` の手順で新規作成する。

**レガシーからの抽出手順**:

1. Phase A-0 で取り込み方針が「リライト」と決まった既存資産から用語を抽出
2. 既存コードから `Grep` で主要なドメイン用語・フィールド名を抽出（変数名・クラス名・DBカラム名）
3. 最初の **10〜20 語程度**で start-small。完全な網羅は目指さない

完了チェックリスト:
- [ ] 最低 10 語のドメイン用語が登録されている
- [ ] フィールド命名規則が 3 項目以上記載されている
- [ ] 既存資産の「リライト」対象から取り込むべき用語が反映されている

### Phase A-3 — OpenAPI 雛形作成

`detailed-design/api/openapi.yaml`（A-1 で合意したパス）を `references/openapi.md` の手順で新規作成する。
フレームワーク別のルート抽出パターンは `references/openapi.md` の「作成手順」節を参照する。

完了チェックリスト:
- [ ] 既存 API ルートがすべて `paths` に列挙されている（フィールド詳細は後回しで可）
- [ ] `tags` で URL プレフィックス単位にグループ分けされている
- [ ] `info.title` `info.version` `servers` が記載されている
- [ ] Docusaurus 上で OpenAPI 仕様にリンクできる（または参照できる）

## Phase B — 推奨セット（AskUserQuestion で選択）

Phase A 完了後、ユーザーに「この時点で合わせて作成するドキュメント」を `multiSelect: true` の AskUserQuestion で確認する。

### AskUserQuestion 例

```
質問: bootstrap と合わせて作成するドキュメントを選んでください（複数選択可）
[基本設計レベル]
- システム構成図 / アーキテクチャ概要
- 機能一覧 / ユースケース一覧
- 画面一覧・画面遷移図
- 非機能要件
- 外部インターフェース一覧
[詳細設計レベル]
- ER図 / テーブル定義
- バッチ設計 / ジョブ一覧
- ADR 初期エントリ（現行アーキテクチャ前提の遡及記録）
```

### 配置（既定）

| ドキュメント | 配置 | 委譲先 reference |
|---|---|---|
| システム構成図（C4 Context / Container） | `basic-design/architecture.md`（+ 必要に応じて `basic-design/architecture/*.drawio`） | `references/architecture.md` |
| 機能一覧 | `basic-design/features.md` | `references/general.md` |
| 画面一覧・画面遷移図 | `basic-design/screens-overview.md` | `references/general.md` |
| 非機能要件 | `basic-design/non-functional-requirements.md` | `references/general.md` |
| 外部インターフェース一覧 | `basic-design/external-interfaces.md` | `references/general.md` |
| ER図 / テーブル定義 | `detailed-design/database.md` | `references/general.md` |
| バッチ設計 | `detailed-design/batch.md` | `references/general.md` |
| ADR 初期エントリ | `basic-design/adr/0001-*.md` | `references/adr.md` |

### 選択されなかったドキュメントの扱い

`docs/README.md` に「未整備ドキュメント一覧」としてチェックボックス形式で記録する:

```markdown
## 未整備ドキュメント一覧

以下は bootstrap 時に作成を見送ったドキュメント。必要に応じて追加する。
後で `/docusaurus-docs` を再実行すると、ここの未チェック項目が再提案される。

- [ ] システム構成図（`basic-design/architecture.md`）
- [ ] ER図（`detailed-design/database.md`）
- [ ] バッチ設計（`detailed-design/batch.md`）
```

### 完了チェックリスト

- [ ] AskUserQuestion で選択されたドキュメントが全て作成されている
- [ ] 各ドキュメントの用語がユビキタス言語定義に準拠している
- [ ] 未選択項目が `docs/README.md` にチェックボックスで記録されている
- [ ] `npm run build` が成功する

## Phase C — 通常フローへ復帰

Phase A / B 完了後、元々のユーザー依頼（例:「〇〇画面の画面定義を作って」）に戻り、
`SKILL.md` のルーティング表に従って通常のドキュメント作成フローを実行する。

このとき、A-1 で合意したディレクトリ構成・A-2 のユビキタス言語・A-3 の OpenAPI が揃っているため、
`references/screen-doc.md` `references/sequence.md` などが前提としていた
「ユビキタス言語参照」「OpenAPI 参照」が正しく機能する。

---

## 再実行時のフロー

`/docusaurus-docs` が**2回目以降に呼ばれた**とき、bootstrap は以下の手順で再実行される。

1. **完了判定** — Phase 0 / A-1 / A-2 / A-3 の成果物がすべて揃っているか確認
   - すべて揃っていれば bootstrap をスキップして通常フロー（Phase C 相当）へ
2. **未整備一覧の読み取り** — `docs/README.md` の「未整備ドキュメント一覧」を Read
3. **未チェック項目の再提案** — 未チェック項目のみを Phase B の AskUserQuestion に再提示
4. **ユーザーが選択したものを作成** — 作成完了した項目は `docs/README.md` のチェックボックスを `[x]` に更新

### 部分的な不整合がある場合

- **ユビキタス言語だけ無い** → Phase A-2 だけ実行
- **OpenAPI だけ無い** → Phase A-3 だけ実行
- **`_category_.json` だけ無い** → Phase A-1 の該当部分だけ補修

各ケースで「不足を検知したら、そのフェーズだけを再実行する」のが原則。**毎回 Phase 0 からやり直さない**。

---

## 原則

- **ユーザーに確認せずディレクトリ構成を決めない**
- **既存ドキュメント資産をまず確認する**（Phase A-0 を飛ばさない）
- **完全性より start-small**（初期エントリは最小限、後から増やす前提で良い）
- **レガシーコードから事実を抽出する**（想像で書かず、`Grep` で既存実装から用語・ルートを拾う）
- **既存に部分的な構成がある場合はそれを尊重する**（無理に提案構成へ寄せない。ユーザーに判断を委ねる）
- **A-2 と A-3 は並列でよい**（直列に硬く実行しない）
- **再実行時は差分だけ補修する**（毎回全フェーズをやり直さない）
