# 画面定義ドキュメント フォーマット定義

対象: ドキュメントルート配下で `*.mdx` を含むディレクトリ
- ブートストラップで合意済みの場合はそのパス（既定は `detailed-design/screens/`）
- ディレクトリ構成が未確立の場合は、先に `references/bootstrap.md` を実行すること
- OpenAPI 定義が存在しない場合は、画面定義を書く前に `references/openapi.md` を先に実行して API 仕様の雛形を作る（画面定義の「API連携」テーブルが OpenAPI を参照する前提のため）

## 必須5セクション構成

```markdown
---
sidebar_position: <番号>
title: "<画面名>"
---

# <画面名>

## 概要

[目的・アクセス権限・前後の遷移 — 1〜3文]

## ワイヤーフレーム

import Drawio from '@theme/Drawio';
import wireframe from '!!raw-loader!../../screens/<ファイル名>.drawio';

<Drawio content=\{wireframe\} />

## 操作フロー

- [操作1]: [結果1]
- [操作2]: [結果2]
- [ダイアログ・モーダルがある場合はここに記述]

## API連携

| 操作 | エンドポイント | メソッド | タイミング |
|---|---|---|---|
| [操作名] | [パス] | [GET/POST/PUT/DELETE] | [画面表示時 / ボタン押下時 etc.] |

※ リクエスト/レスポンスフィールドの詳細は OpenAPI 仕様（`detailed-design/api/openapi.yaml` 等）を参照。OpenAPI が存在しない場合は `references/openapi.md` を先に実行して雛形を作成する。

## UX仕様

- [ローディング方針]
- [エラーコード] → [表示方式]
- [バリデーション方針（方針レベル）]
```

## 各セクションの書き方

### 概要
- 「誰が」「何のために使う画面か」「どこから来てどこへ行くか」を1〜3文で書く
- 権限（管理者のみ / 認証済みユーザーのみ）を明記する

### 操作フロー
- 「〜したとき、〜になる」の粒度で書く
- ダイアログ・モーダルの存在と開閉条件を記述する
- 成功・失敗の分岐を書く

### API連携
- 「タイミング」は「画面表示時」「ボタン押下時」「入力変更時」のように具体的に書く
- フィールドの詳細は OpenAPI 仕様に任せる

### UX仕様
- エラーはコード名（`UNAUTHORIZED` 等）で参照し、文言は書かない
- ローディング: スケルトン / スピナー / ボタン無効化 のどれかを明示する
- バリデーションは「方針」のみ（「必須項目は送信前にチェック」レベル）

## 書いてはいけないもの

- コンポーネント名・型定義・変数名（コードで管理）
- APIレスポンスのフィールド詳細（OpenAPI 仕様で管理）
- 具体的なエラーメッセージ文言（エラーコードで参照）
- フォームバリデーションの詳細テーブル

## 判断基準

> 「このコード行が変わったとき、このドキュメントの記述も必ず更新が必要か？」
> - YES → 書かない（コードと二重管理になる）
> - NO → 書いてよい（設計意図・方針レベル）

## ファイル配置

- 画面一覧: 画面定義ディレクトリ内の `overview.md`
- drawio ファイル: ドキュメントルート配下で `*.drawio` を含むディレクトリ（通常 `screens/`）
