# Docusaurus ドキュメント更新スキル

## 役割

ユーザーの依頼からドキュメント種別を判定し、対応する手順ファイルを読んで実行する。
プロジェクト構造は既存のファイル・ディレクトリから直接読み取る。

## 共通ルール（全種別に適用）

- ドキュメントは**日本語**で記述する
- **各ドキュメントの冒頭に Docusaurus フロントマター（`title` / `sidebar_position`）を付ける**
  - 対象は `.md` / `.mdx` 全般（`.drawio` / `.yaml` 等のバイナリ・データ系は除く）
  - 値の決め方は後述「フロントマター採番ルール」に従う
  - フロントマター欠落は Docusaurus ビルドの警告・サイドバー並びの不整合の原因になるため必須とする
- **設計意図・方針を書き、実装詳細は書かない**
  - 書かない対象: コンポーネント名・関数名・変数名・型定義・アルゴリズム名・内部パラメータ・エラーメッセージ文言
  - 書く対象: 設計判断の理由、処理フローの意図、用語の定義、操作の目的、ADR の経緯
  - 判断基準: 「このコード行が変わったとき、このドキュメントの記述も必ず更新が必要か？」に YES なら書かない（コードと二重管理になる）
- 用語・フィールド名はプロジェクトのユビキタス言語定義に準拠する
  - ドキュメントルート配下で `ubiquitous-language` を含むファイルを探して参照する
- API に言及する場合は OpenAPI 定義を正として参照する（`**/openapi.{yaml,yml,json}`）
- `docusaurus.config.ts` を新規作成・変更する場合は `assets/docusaurus-config-example.ts` を参考にする
- **Markdown モード（`.md`）と MDX モード（`.mdx`）を区別する**
  - `docusaurus.config.ts` の `markdown.format` が `'md'` に設定されているか、対象ファイルの拡張子が `.md` の場合は**プレーン Markdown モード**として扱う
  - プレーン Markdown モードでは `{}` や `<>` のエスケープ・前処理を行ってはならない（コードスパン内に意図しないバックスラッシュが混入し、`{#anchor-id}` 形式の見出し ID も破壊される）
  - MDX モード（`.mdx` ファイル）でのみ JSX 互換の記法に注意し、未エスケープの `<` `>` に気をつける
  - 設定を生成・変更する際は**まず現在どちらのモードか**を確認してから作業する
- 完了前にビルドを確認する。Glob でスキルの `scripts/build-check.sh` の**絶対パス**を取得し、`bash <絶対パス>` で実行する（作業ディレクトリのずれを防ぐため、`cd` は必ず絶対パスで行うこと）
- ビルド結果をユーザーに報告してから完了宣言する

## 判定のフロー

1. ドキュメントルートを特定する（後述「プロジェクト構造の探索」に従う）
2. **ブートストラップ必要性の判定**（後述「ブートストラップ判定」に従う）— 不足があれば `references/bootstrap.md` を先に実行してから本来の依頼に戻る
3. ユーザーの依頼文を分析してキーワードを抽出する
4. ルーティング表で種別を決定する（複数該当する場合は最も具体的なものを選ぶ）
5. 種別の「前提条件」が不足していないか確認する。不足していれば該当の前提手順を先に実行する
6. 対応する手順ファイルを Read する
7. 既存ドキュメントのフォーマットを参考にしながら作業を実行する

## 種別が曖昧な場合

「ドキュメントを更新して」など種別が特定できない場合は、対象ファイルをユーザーに確認してから判定する。

## フロントマター採番ルール

全 `.md` / `.mdx` ドキュメントに以下のフロントマターを必ず付ける。

```markdown
---
sidebar_position: <番号>
title: "<表示タイトル>"
---
```

### `title`

- 各 reference のテンプレートに記載された値を使う（例: `"データベース設計"` / `"エラーコード一覧"` / `"アーキテクチャ設計"`）
- 画面定義・ADR のような**動的に決まるタイトル**は、各 reference のテンプレートの指示に従う（画面名・ADR 番号を埋める）

### `sidebar_position` の採番手順

1. **配置先ディレクトリを確認する**（例: `docs/basic-design/`、`docs/detailed-design/`、`docs/detailed-design/screens/` 等）
2. **同ディレクトリ内の既存 `.md` / `.mdx` の `sidebar_position` を Grep で収集**
   ```bash
   grep -r "sidebar_position:" <配置先ディレクトリ> --include="*.md" --include="*.mdx"
   ```
3. **既存最大値 + 1 を採用する**（番号の衝突を避ける）
   - 例: 既存が `1, 2, 3` なら新規は `4`
   - 既存が空（初回ファイル）なら `1` から開始
4. **`_category_.json` の `position` は別空間**なので気にしなくてよい（カテゴリ間の並び順と、カテゴリ内のファイル並び順は独立している）

### 既存ファイルを更新する場合

- 既存のフロントマターが欠落している場合は採番手順に従って**追加する**（既存内容の破壊的変更と見なさない — ビルド挙動の整合性のため）
- 既存のフロントマターがある場合は**触らない**（並び順の意図を尊重する）

## プロジェクト構造の探索

### ドキュメントルートの特定

以下の優先順で探索し、最初に見つかった方法で確定する：

1. **自動検出**: プロジェクトルートから `docusaurus.config.js` または `docusaurus.config.ts` を探す。  
   見つかったら、その設定ファイルと同階層の `docs/` をドキュメントルートとする。
2. **CLAUDE.md から読み取る**: プロジェクトの CLAUDE.md にドキュメントルートの記載がないか確認する。
3. **ユーザーに確認**: 上記で特定できない場合はユーザーにドキュメントルートのパスを確認する。

### ブートストラップ判定

以下を順に確認する。**1 つでも不足していればブートストラップを発動**する（`references/bootstrap.md` を Read して実行）:

0. **Docusaurus 導入**: `docusaurus.config.js` または `docusaurus.config.ts` が存在するか  
   → 存在しない場合は `references/bootstrap.md` の Phase 0 から実行する（ドキュメントルートが未確立なので、上記「ドキュメントルートの特定」も bootstrap の中で決まる）
1. `basic-design/` と `detailed-design/` ディレクトリが両方存在するか
2. `ubiquitous-language` を名前に含むファイルが存在するか
3. OpenAPI 定義（`**/openapi.{yaml,yml,json}` または相当のファイル）が存在するか

上記がすべて揃っている場合はブートストラップをスキップして通常フローに進む。

**重要**: ディレクトリ構成（`basic-design/` / `detailed-design/` の 2 カテゴリ構成で良いか等）は**必ずユーザーに確認してから決定する**。エージェントが独自ルールで構成を決めてはならない。

### ドキュメント種別ごとの探索

ドキュメントルート配下で以下を探す：

- **画面定義**: `*.mdx` ファイルを含むディレクトリ（通常 `screens/` や `detailed-design/screens/`）
- **drawio**: `*.drawio` ファイルを含むディレクトリ
- **ADR**: `adr/` や `adr-*.md` を含むディレクトリ
- **シーケンス図**: `sequence.md` や `sequence-*.md` を探す
- **ユビキタス言語**: `ubiquitous-language.md` を探す
- **OpenAPI 定義**: `**/openapi.{yaml,yml,json}` を探す（通常 `detailed-design/api/`）
- **データベース設計**: `database.md` / `er.md` / `schema.md` 等を探す（通常 `detailed-design/database.md`）
- **非機能要件**: `non-functional-requirements.md` / `nfr.md` を探す（通常 `basic-design/`）
- **エラーコード管理**: `error-codes.md` / `errors.md` を探す（通常 `detailed-design/`）
- **アーキテクチャ設計**: `architecture.md` を探す（通常 `basic-design/architecture.md`）。`basic-design/architecture/` ディレクトリに `.drawio` が置かれる場合もある
- **既存ドキュメントの書式**: 同種の既存ファイルを Read してフォーマットを踏襲する
